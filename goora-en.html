Goora

Product Designer • WebApp • 2025

Designed the core flow of a taxi app: from finding a car to rating a completed ride. Built a clean screen structure so users clearly understand their order status at every stage of the trip.

Task

I had to design a web interface for ordering a taxi as part of a under a tight deadline. I needed to manually build the full component system, set up auto layouts, and define the logic of the main screen.

I decided to work through not only the main screen, but also the key service states: driver search, driver found, ride in progress, and ride completion.

Core flow

Main screen

I split the interface into two parts: the order panel on the left and the map on the right. In the panel, the user enters the route, selects a fare, chooses a payment method, and confirms the order. The map shows the city, nearby cars, and helps users immediately understand the trip context.

The app detects the user’s geolocation and shows the city they are currently in. Even before entering an address, the user sees an approximate fare price, so they can understand the cost in advance.

Route input

Before typing, the user sees quick actions: select a point on the map or open saved addresses. This is useful when the user does not want to enter an address manually.

When the user starts typing, address suggestions move to the foreground. Quick actions remain below and become secondary.

Trip setup

After selecting the addresses, the user sees the assembled order: route, estimated trip time, fare options, payment, bonuses, and price.

The “From” and “To” points are shown on the map with address labels. This helps the user visually check the route before ordering and be confident that the trip is being booked to the right locations.

I added the ability to select several fare options at once. Not all users may be familiar with this mechanic, so selecting one fare does not disable the other options. The number of selected fares is also shown on the right — hinting that more than one option can be chosen.

The price is shown as a range because the order can be sent to several fare categories at once. The final price depends on which driver accepts the order first.

Driver search

A clear status bar appears at the top of the screen. It shows that the app is currently looking for a car. This is important because the user immediately sees that the app has responded and does not have to worry about the order.

Order information is reduced to the key parameters: route, selected fares, payment method, and price. The user sees the conditions under which the system is searching for a car.

The pickup point becomes unavailable for editing because the search has already started from a specific address. At the same time, the user can still change the destination and add more stops.

The cancel button remains available, but does not pull attention away from the main flow.

Driver found

At this stage, the focus shifts to the car and its arrival time. The status bar updates — the user sees that the car is already on its way and understands how many minutes are left to wait.

A driver card appears in the panel with the key details: name, rating, number of rides, car model, color, and plate number. Quick actions are also available — call or message the driver.

Once the driver has been found, the addresses can no longer be changed. The order has been accepted, and the driver is following the selected route.

The fare and price also become final: instead of a range, the interface shows the specific fare and final price.

The driver’s car is shown on the map. This helps the user understand where the driver is right now and how they are moving toward the pickup point.

In progress

The status bar shows that the ride has started and displays the remaining time to the destination. The map shows the car’s current position so the user can check whether the ride is following the route.

All important information remains in the panel: driver, route, fare, payment method, and price. At this stage, the user is no longer configuring the order, but tracking the ride.

There is a “Report an issue” button at the bottom. It does not interfere with the main flow, but remains available in case something goes wrong.

Ride completion

I kept only what is necessary here: driver information, trip price, rating, comment field, and the button for submitting feedback.

If the user does not want to leave feedback, they can immediately start a new ride.

Outcome

The project was created under a tight deadline, and it was important for me to deliver a high-quality product. Even though the company I made this assignment for never even looked at the result :), I completed a genuinely interesting task, fulfilled all the requirements, and gained valuable experience.
